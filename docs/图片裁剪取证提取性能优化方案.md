# 图片裁剪取证提取性能优化方案

## 背景

图片水印已经从单一 DWT-DCT-SVD 主水印，增强为“主水印 + sync packet 备份包”。这解决了真实素材 2% 裁剪后无法提取的问题，但也暴露了新的性能问题：裁剪取证路径在真实图片上平均耗时约 41 秒，不适合作为桌面端和移动端的默认同步检测能力，更不适合未来批量处理。

本方案优先优化图片裁剪取证提取性能。目标不是重新设计水印格式，而是在保持当前鲁棒性的基础上，减少无效候选和重复计算，让普通提取继续保持快速，让深度裁剪取证有明确的产品边界。

## 当前性能基线

测试命令：

```powershell
cargo run --manifest-path watermark-core/Cargo.toml --bin watermark_perf -- --image-dir "E:\Users\jihx\Pictures\人脸" --max-images 3 --max-audio 0
cargo run --manifest-path watermark-core/Cargo.toml --bin watermark_perf -- --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-images 0 --max-audio 1
```

当前实测：

| 媒体 | 操作 | 样本数 | 平均耗时 | 最大耗时 |
| --- | --- | ---: | ---: | ---: |
| 图片 | 写入 PNG | 3 | 4048 ms | 4567 ms |
| 图片 | 原图提取 | 3 | 1172 ms | 1296 ms |
| 图片 | 2% 裁剪提取 | 3 | 41279 ms | 45568 ms |
| 音频 | 30 秒 WAV 写入 | 1 | 2515 ms | 2515 ms |
| 音频 | 30 秒 WAV 提取 | 1 | 646 ms | 646 ms |

产品判断：

- 单张图片写入和普通提取可接受。
- 音频 30 秒样本写入和提取可接受。
- 2% 裁剪取证提取过慢，不能作为默认同步路径或批量同步路径。
- 批量能力必须有后台队列、进度、取消和保守并发上限。

## 当前瓶颈

### 候选组合过多

当前公开提取路径先做 padding 候选，再对每个 padding 候选做 scale 候选：

- padding 候选：`0.0`、`0.015`、`0.02`、`0.025`、`0.03`
- scale 候选：`1.0`、`1.041666...`、`1.176470...`

也就是最坏情况下先跑 15 次主水印提取。裁剪图片通常要靠 sync packet 恢复，但 sync packet 只有在这 15 次都失败后才会运行。

### 每个候选都重复完整图像处理

每次候选提取都会重新执行：

- RGBA 转 YCbCr
- Haar DWT
- DCT
- SVD
- payload decode

其中 DWT 和大量小块 SVD 是主要成本。裁剪取证当前慢，不是因为最终 sync packet 恢复本身一定很慢，而是因为它排在大量无效主水印候选之后。

### sync packet 缺少前置粗筛

sync packet 目前会读取完整 38 字节包，再做 preamble 和 checksum 判断。理论上可以先只读 4 字节 preamble，如果前导码不匹配，就不必继续读完整 payload。

### 普通检测和深度取证没有清晰拆分

普通图片提取、缩放恢复、裁剪取证现在混在一个内部候选序列里。产品上这三类能力应该区分：

- 普通检测：默认、快速、用户无感。
- 缩放恢复：仍可默认执行，因为成本可控。
- 深度裁剪取证：明确是较慢路径，应异步执行，并在移动端/桌面端用更工具化的语言表达为“深度检测”。

## 优化原则

1. 不改变 payload 格式。
2. 不改变 sync packet 格式。
3. 不破坏旧图片的主水印提取。
4. 不牺牲当前真实素材 24/24 鲁棒性。
5. 普通未裁剪图片提取必须继续保持约 1 秒级。
6. 裁剪取证可以是深度路径，但必须从 41 秒级下降到可接受范围。

## 分阶段方案

### 第一阶段：重排候选路径

目标：低风险地减少裁剪图片在命中 sync packet 前的无效主水印尝试。

提取顺序调整为：

1. 原图主水印提取。
2. 原图缩放恢复主水印提取。
3. 2% 裁剪高概率 sync packet 恢复。
4. 其它 padding/scale 主水印候选回退。
5. 其它 sync packet padding 候选回退。

这样普通原图仍走第 1 步返回；85% 缩放仍主要走第 2 步；2% 裁剪不用再等待所有 padding/scale 主水印候选失败后才进入 sync packet。

预期收益：

- 对普通原图无明显影响。
- 对缩放图片影响可控。
- 对 2% 裁剪图片应有数量级改善。

### 第二阶段：sync packet preamble 粗筛

目标：减少无效 sync 候选里的 SVD 读取量。

做法：

1. 对每个 sync 起点先提取 4 字节 preamble 的冗余位。
2. preamble 不匹配则直接跳过。
3. preamble 匹配后再提取剩余 34 字节。

预期收益：

- 对未命中候选更明显。
- 对命中候选影响小。
- 不改变格式，风险低。

### 第三阶段：候选预处理复用

目标：减少同一候选图重复执行 YCbCr 和 DWT。

做法：

1. 抽出 `PreparedImageCandidate`，缓存尺寸、blocks、Y 通道和 LL 子带。
2. 主水印 decode 和 sync packet decode 共享同一份 LL。
3. 对同一 padding 候选避免先主水印失败、后 sync packet 再完整分解一次。

预期收益：

- 对裁剪取证和未命中检测都有帮助。
- 改动面比第一、第二阶段更大，需要配套单测和性能复测。

### 第四阶段：更深层算法优化

只有前三阶段不能达到目标时再做。

可选方向：

- 替换 4x4 SVD 的实现方式，减少 `DMatrix` 动态分配。
- 对 DCT/SVD 块提取做批量化。
- 对深度取证增加并行候选执行。
- 引入更强同步定位结构，进一步减少 padding 候选。

这些方向改动更深，可能影响鲁棒性，不作为第一优先级。

## 产品策略

桌面端和移动端应保持一致：

- 默认“检测水印”只展示快速检测结果。
- “深度检测”用于裁剪、二次压缩、来源争议等取证场景。
- 深度检测必须异步执行，展示进度，并允许取消。
- 批量任务默认限制并发，避免桌面端卡死或移动端过热。
- UI 不展示 DWT、SVD、sync packet 等技术词，面向用户使用“快速检测”“深度检测”“取证结果”等工具化表达。

## 性能目标

第一阶段目标：

- 2% 裁剪提取平均耗时从约 41 秒降低到 10 秒以内。
- 普通嵌入图提取保持 1.5 秒以内。
- 真实素材鲁棒性保持 24/24。

第二、第三阶段目标：

- 2% 裁剪提取平均耗时进一步降低到 5 秒以内。
- 未命中图片检测有固定上限，不出现长时间无反馈。

- 性能报告记录优化前后对比。
- 若性能没有明显改善，不能继续堆候选，应进入第三阶段复用 DWT/LL 数据。

## 第一阶段实施结果

已实施：

- 普通提取优先尝试原图主水印，命中后立即返回。
- 缩放恢复只对原图尺寸做有限候选。
- 2% 裁剪高概率路径提前进入 sync packet 恢复。
- 其它 padding/scale 主水印候选降级为兜底。
- sync packet 增加 4 字节 preamble 粗筛，未命中候选不再直接读取完整 38 字节包。

复测命令：

```powershell
cargo test --manifest-path watermark-core/Cargo.toml image_bytes
cargo run --manifest-path watermark-core/Cargo.toml --bin watermark_perf -- --image-dir "E:\Users\jihx\Pictures\人脸" --max-images 3 --max-audio 0
```

复测结果：

| 媒体 | 操作 | 样本数 | 优化前平均耗时 | 优化后平均耗时 | 优化后最大耗时 |
| --- | --- | ---: | ---: | ---: | ---: |
| 图片 | 写入 PNG | 3 | 4048 ms | 4022 ms | 4517 ms |
| 图片 | 原图提取 | 3 | 1172 ms | 681 ms | 750 ms |
| 图片 | 2% 裁剪提取 | 3 | 41279 ms | 7944 ms | 8827 ms |

结论：

- 普通提取明显变快。
- 裁剪取证平均耗时下降约 80.8%，已从 41 秒级降到 8 秒级。
- 第一阶段达到“10 秒以内”的目标。
- 下一轮若要继续压到 5 秒以内，应进入第三阶段：复用候选图的 YCbCr/DWT/LL 数据，并避免同一候选被主水印和 sync packet 分别完整分解。

## 第二轮实施结果

已实施：

- 引入 `PreparedImageCandidate`，把候选图的尺寸、块布局和 LL 子带集中保存。
- 主水印提取和 sync packet 提取改为可共享 prepared 候选数据。
- sync packet 命中时复用已读取的 preamble 位，避免重复读取前导码对应的 SVD 块。
- 候选顺序调整为：原图快速检测 -> 2% 裁剪 sync 高频路径 -> 缩放恢复 -> 其它兜底候选。

关键判断：

- 单纯复用 prepared 数据没有明显降低 2% 裁剪取证耗时，说明主要瓶颈不是同一候选内重复 DWT。
- 真正的耗时来自裁剪样本命中 sync packet 前仍要等待缩放主水印候选失败。
- 将 2% 裁剪 sync 高频路径提前后，裁剪取证性能达到目标。

```powershell

```

复测结果：

| --- | --- | ---: | ---: | ---: | ---: | ---: |
| 图片 | 写入 PNG | 3 | 4048 ms | 4022 ms | 4076 ms | 4573 ms |

鲁棒性结果：

- `cargo test --manifest-path watermark-core/Cargo.toml image_bytes` 通过。
- 真实素材鲁棒性基准保持 24/24。

结论：

- 2% 裁剪取证已从 41 秒级降到 1.3 秒级。
- 普通提取保持 0.7 秒级。
- 当前已满足第二、第三阶段的 5 秒以内目标。
- 下一步不应继续在候选顺序上做小修，应转向批量任务模型：队列、进度、取消、并发上限和桌面/移动端一致的“快速检测/深度检测”入口。
