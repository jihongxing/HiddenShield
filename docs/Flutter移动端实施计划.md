# HiddenShield Flutter 移动端实施计划

**目标**：把现有桌面端能力落到 iOS / Android，并尽量保持和桌面端同构。  
**前提**：当前没有统一用户体系，先不强行上账号系统。

---

## 1. 总目标

移动端先做三件事：

1. 本地图片盲水印嵌入 / 提取。
2. 本地音频盲水印嵌入 / 提取。
3. 与桌面端同步版权库元数据和派生链。

不做的事：

- 不做本地视频盲水印。
- 不做移动端 FFmpeg。
- 不做多账号、多租户、团队协作。

---

## 2. 身份策略

### 2.1 现状判断

当前桌面端和移动端都没有统一账号体系。如果直接上登录，会把产品最关键的本地能力拖慢。

### 2.2 推荐方案

采用三层身份，不引入强制登录：

- `creator identity`：创作者身份，跨设备共享。
- `device identity`：设备身份，单机唯一。
- `sync identity`：同步配对身份，仅用于桌面互联。

### 2.3 落地方式

- 桌面端首次初始化时生成 `creator identity`。
- 桌面端可导出加密后的创作者身份包，手机端扫码或导入文件接收。
- 手机端也允许独立初始化，再通过桌面配对把身份合并到同一创作者名下。
- 设备身份永远独立，不能共享。

### 2.4 规则

- 水印 UID 仍由创作者身份 + 设备身份生成。
- 同一创作者在不同设备上的 UID 可以不同，但归属同一创作者档案。
- 同步只认配对关系，不依赖账号密码。

## 3. 实施顺序

### Phase -1：身份契约冻结，3-5 天

目标：先把跨端身份和数据字段定死。这是移动工程骨架之前的前置闸门，不替代设计方案里的 Phase 0。

交付物：

- 创作者身份导入 / 导出格式。
- 设备身份规则。
- 同步配对规则。
- `vault_records` 字段对齐表。

验收：

- 桌面端和移动端都能解释同一套创作者身份。
- 不需要账号也能启动产品。

### Phase 0：Flutter 工程骨架，3-5 天

目标：搭起 App 外壳和 FFI 通道。

交付物：

- Flutter 工程。
- `flutter_rust_bridge` 接入。
- iOS / Android 构建链跑通。
- 四个主 Tab：工作台、取证、版权库、设置。

验收：

- 真机可启动。
- Rust 核心能被 Flutter 调用。

### Phase 1：图片 MVP，2 周

目标：先把最核心闭环做出来。

功能：

- 图片导入。
- 图片嵌入。
- 图片提取。
- 版权库写入。
- 结果页展示。

验收：

- 手机能完成第一条图片版权记录。

### Phase 2：图片取证，1 周

功能：

- 选择疑似侵权图片。
- 提取水印。
- 匹配本地版权库。
- 复制摘要 / 分享结果。

验收：

- 能现场完成图片取证闭环。

### Phase 3：音频 MVP，1-2 周

功能：

- WAV 导入。
- WAV 嵌入。
- WAV 提取。
- 音频记录入库。

约束：

- 只承诺 WAV。
- 不在移动端做通用格式转码。

验收：

- iOS / Android 都能稳定处理 WAV。

### Phase 4：桌面配对与同步，2-4 周

功能：

- 桌面扫码配对。
- 版权库元数据同步。
- 派生链同步。
- 冲突标记。

同步范围：

- 同步记录。
- 不默认同步原始媒体文件。

验收：

- 一台手机可以和一台桌面完成双向同步。

### Phase 5：稳定性收尾，1-2 周

功能：

- 离线补发。
- 错误码统一。
- 权限与隐私收口。
- 导出 / 删除 / 恢复语义确认。

验收：

- 崩溃率、同步失败率、取证失败率都能被观测。

---

## 4. 产品结构

移动端和桌面端保持同名能力区：

- 工作台：图片 / 音频创建入口。
- 取证：图片 / 音频提取与报告。
- 版权库：时间线、详情、链路。
- 设置：身份、同步、隐私、权益、帮助。

不要让移动端出现新的概念名，避免用户和研发两头切换语义。

---

## 5. 技术约束

- Flutter 负责界面、权限、文件选择、保存。
- Rust 负责水印算法、payload 编解码、哈希。
- SQLite 负责本地版权库。
- 同步层只处理元数据和链路，不碰原始媒体内容。

---

## 6. 风险与回退

| 风险 | 影响 | 回退 |
|------|------|------|
| 没有统一账号 | 跨端身份混乱 | 先用创作者身份包 + 桌面配对 |
| FFI 打包复杂 | 首版延期 | 先只做图片桥接，再补音频 |
| 音频格式兼容差 | 用户导入失败 | 首版只支持 WAV |
| 同步冲突复杂 | 数据不一致 | 先标记冲突，不自动合并 |

---

## 7. 第一周任务

1. 冻结身份契约。
2. 确定 Flutter 工程目录。
3. 搭 `flutter_rust_bridge` 最小样例。
4. 做一个图片嵌入测试页。
5. 做一个创作者身份导入 / 导出草案。

---

## 8. 结论

这条路线的核心是：**先统一身份语义，再统一功能体验**。  
没有账号系统也能落地，而且不会把移动端拖进一套重型认证系统里。
