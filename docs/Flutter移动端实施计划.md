# HiddenShield Flutter 移动端实施计划

**目标**：把现有桌面端能力落到 iOS / Android，并尽量保持和桌面端同构。  
**前提修订**：当前“移动端直连桌面端同步”不再作为正式产品同步方案。正式跨端同步应基于统一账户和云同步服务；局域网桌面直连仅保留为开发联调 / 高级调试能力。详见 `docs/统一账户与云同步重设计方案.md`。

---

## 1. 总目标

移动端先做三件事：

1. 本地图片盲水印嵌入 / 提取。
2. 本地音频盲水印嵌入 / 提取。
3. 本地保存版权库元数据和派生链；跨端同步改为后续统一账户 + 云同步。

不做的事：

- 不做本地视频盲水印。
- 不做移动端 FFmpeg。
- 不做多账号、多租户、团队协作。

---

## 2. 身份策略

### 2.1 现状判断

当前桌面端和移动端都没有统一账号体系。早期可以不强制登录以保护本地能力体验，但只要进入跨设备同步，就必须引入统一账户。

### 2.2 推荐方案

本地模式采用三层身份，不引入强制登录：

- `creator identity`：创作者身份，跨设备共享。
- `device identity`：设备身份，单机唯一。
- `sync identity`：同步身份。正式方案应是账户云同步身份；桌面互联只作为局域网调试身份。
- `entitlement identity`：账户权益身份，移动端和桌面端共享同一份订阅状态。

### 2.3 落地方式

- 桌面端和移动端首次使用时都可以生成本机 `creator identity`。
- 继续使用 HiddenShield 账户后，系统自动判断账号是否存在；新用户自动创建账户，老用户直接进入，并把本机创作者身份加入账户下的默认创作者档案。
- 云同步开启后，默认创作者档案会在移动端和桌面端同步。
- 局域网桌面互联只保留为高级调试 / 临时迁移，不作为正式身份合并路径。
- 设备身份永远独立，不能共享。

### 2.4 规则

- 水印 UID 仍由创作者身份 + 设备身份生成。
- 同一创作者在不同设备上的 UID 可以不同，但归属同一创作者档案。
- 正式同步只认账户、设备注册和创作者档案关系。
- 局域网调试只用于工程验证，不作为正式归属裁判。

## 3. 实施顺序

### Phase -1：身份契约冻结，3-5 天

目标：先把跨端身份和数据字段定死。这是移动工程骨架之前的前置闸门，不替代设计方案里的 Phase 0。

交付物：

- 创作者身份导入 / 导出格式。
- 设备身份规则。
- 同步配对规则。
- `vault_records` 字段对齐表。

验收：

- 桌面端和移动端都能解释同一套创作者身份。
- 不需要账号也能启动产品。

### Phase 0：Flutter 工程骨架，3-5 天

目标：搭起 App 外壳和 FFI 通道。

交付物：

- Flutter 工程。
- `flutter_rust_bridge` 接入。
- iOS / Android 构建链跑通。
- 四个主 Tab：工作台、取证、版权库、设置。

验收：

- 真机可启动。
- Rust 核心能被 Flutter 调用。

### Phase 1：图片 MVP，2 周

目标：先把最核心闭环做出来。

功能：

- 图片导入。
- 图片嵌入。
- 图片提取。
- 版权库写入。
- 结果页展示。

验收：

- 手机能完成第一条图片版权记录。

### Phase 2：图片取证，1 周

功能：

- 选择疑似侵权图片。
- 提取水印。
- 匹配本地版权库。
- 复制摘要 / 分享结果。

验收：

- 能现场完成图片取证闭环。

### Phase 3：音频 MVP，1-2 周

功能：

- WAV 导入。
- WAV 嵌入。
- WAV 提取。
- 音频记录入库。

约束：

- 只承诺 WAV。
- 不在移动端做通用格式转码。

验收：

- iOS / Android 都能稳定处理 WAV。

### Phase 4：统一账户与云同步，4-8 周

#### Phase 4A：本地账户契约，已完成

功能：

- 账户继续态占位。
- 默认工作区占位。
- 设备注册占位。
- 默认创作者档案占位。
- 订阅权益快照占位。

验收：

- 移动端 `SyncProfile` 能持久化账户、工作区、设备、创作者档案和权益快照。
- 设置页能展示这些账户契约字段。
- 不依赖真实后端也能验证本地状态流转。

#### Phase 4B：云同步 API MVP，下一步

功能：

- 真实登录注册合一入口。
- 设备登记随账户继续自动完成。
- 创作者档案 upsert / pull。
- 订阅权益 pull。
- 云端同步队列。
- 版权库元数据同步。
- 派生链同步。
- 自动解决审计。

同步范围：

- 同步版权记录、取证记录、创作者档案、订阅权益和审计记录。
- 默认不同步原始图片、加水印后的图片、原始音频、加水印后的音频、原始视频、加水印后的视频和本地文件路径。

验收：

- 同一账户下，一台手机和一台桌面可以通过云端完成双向同步。

当前移动端进展：

- `CloudAccountClient` 已实现 `POST /v1/auth/continue` 请求 / 响应模型。
- `MobileAppState` 已支持注入真实 `CloudAccountClient`，并把响应映射到 `SyncProfile`。
- `SyncProfile` 已持久化系统云服务地址快照，登录注册合一和云同步推拉共用同一个云服务入口。
- `CloudSyncTransport` 已实现版权库 `POST /v1/sync/events:batch` 和 `GET /v1/sync/changes` client，并覆盖鉴权、设备登记、游标拉取和配置失败测试。
- 仓库根目录已提供最小真实云同步后端和契约验证脚本：`npm run cloud:backend` / `npm run cloud:contract`；旧 mock 仅作为协议对照工具保留。
- 桌面端 Tauri 已新增同协议云同步 client，并暴露 `continue_cloud_account`、`push_desktop_vault_record_to_cloud`、`fetch_cloud_changes`、`flush_desktop_cloud_sync_queue` 命令。
- 桌面端已持久化 `cloud_sync_profile.json`，设置页已提供“账户与云同步”主入口，LAN 配对码服务已移入高级调试区。
- 桌面端已新增本地 `cloud_sync_queue`：新版权记录落库后自动入队，版权库页面可查看待同步 / 失败 / 已同步摘要，并通过“同步队列”批量补发。
- 桌面版权库保留单条记录“同步此记录”动作，并提供“拉取云变更”入口。
- 桌面端拉取云变更会复用现有 `record_sync_event` 自动解决规则落库，并更新 `cloud_sync_profile.json` 的远端游标。
- 移动端和桌面端现在都具备对接真实云同步后端的 client 骨架，下一步应实现真实云端服务的自动解决裁判、设备游标和订阅权益刷新。

说明：

- 原“桌面扫码配对 / 桌面 HTTP 同步”降级为高级调试能力，不作为正式产品主路径。

### Phase 5：稳定性收尾，1-2 周

功能：

- 离线补发。
- 错误码统一。
- 权限与隐私收口。
- 导出 / 删除 / 恢复语义确认。

验收：

- 崩溃率、同步失败率、取证失败率都能被观测。

## 4. 产品结构

移动端和桌面端保持同名能力区：

- 工作台：图片 / 音频创建入口。
- 取证：图片 / 音频提取与报告。
- 版权库：时间线、详情、链路。
- 设置：身份、同步、隐私、权益、帮助。

不要让移动端出现新的概念名，避免用户和研发两头切换语义。

---

## 5. 技术约束

- Flutter 负责界面、权限、文件选择、保存。
- Rust 负责水印算法、payload 编解码、哈希。
- SQLite 负责本地版权库。
- 同步层只处理元数据和链路，不碰原始媒体内容。

---

## 6. 风险与回退

| 风险 | 影响 | 回退 |
|------|------|------|
| 没有统一账号 | 跨端身份混乱 | 本地模式不强制登录；跨端同步必须继续使用账户 |
| 创作者身份不同步 | 双端水印归属不一致 | 继续使用账户后把本机身份加入账户创作者档案 |
| 权益只在单端生效 | 用户体验割裂 | 订阅权益做账户级快照并双端同步 |
| FFI 打包复杂 | 首版延期 | 先只做图片桥接，再补音频 |
| 音频格式兼容差 | 用户导入失败 | 首版只支持 WAV |
| 同步冲突复杂 | 数据不一致 | 云端作为最终裁判，客户端展示自动解决审计 |

## 7. 第一周任务

1. 冻结身份契约。
2. 确定 Flutter 工程目录。
3. 搭 `flutter_rust_bridge` 最小样例。
4. 做一个图片嵌入测试页。
5. 做一个创作者身份导入 / 导出草案。

---

## 8. 结论

这条路线的核心是：**先统一身份语义，再统一功能体验**。  
没有账号系统也能落地，而且不会把移动端拖进一套重型认证系统里。
