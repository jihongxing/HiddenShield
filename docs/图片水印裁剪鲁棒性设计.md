# 图片水印裁剪鲁棒性设计

## 背景

当前图片水印采用 DWT-DCT-SVD 写入 256 bit payload，并用 3 倍冗余做多数投票。它已经能稳定覆盖：

- PNG 重新编码
- JPEG Q90 / Q75
- 85% 缩放
- 常规写入 / 提取 / 拒绝重复写入

真实素材基准里剩余的失败项是 2% 边缘裁剪。根因不是 payload 本身损坏，而是裁剪会改变 DWT 的 LL 子带网格。原有算法从左上角按固定块序读取，裁剪后块序和原写入序列错位，常见表现是 payload magic 解码成全 `FF` 或其它无效头。

## 已验证但不采纳的方案

### 1. 块偏移扫描

做法：提取时尝试 `(0,0)`、`(1,0)`、`(0,1)`、`(1,1)` 等少量块偏移。

结论：能覆盖小尺寸单测，但不能稳定覆盖真实 1080x1080 样本的 2% 裁剪。真实裁剪每边约 21 像素，映射到 LL/DCT 块后不是简单的 1-2 个块偏移。

### 2. 中部锚点副本

做法：除主水印外，在图像中部额外写入一份 payload。

结论：仍不稳定。DWT 不是平移不变的，裁剪后整张图重新分解，原中部锚点不再对应同一批 LL 系数。

### 3. 全图周期铺写 + 全相位恢复

做法：把 payload 冗余位按周期铺满全部 LL 块，提取时扫描 768 个相位。

结论：方向上更接近裁剪问题，但默认写入和提取成本过高。真实素材 benchmark 会超时，不适合作为产品默认路径。

## 正式方案

采用“主水印 + sync packet 备份包”：

1. 主水印层保持现有顺序布局。
   - 继续写入前 `PAYLOAD_BITS * REDUNDANCY` 个块。
   - 保持老图可提取。
   - 保持常规提取性能。

2. sync packet 备份包。
   - payload 业务内容仍为 32 字节。
   - 外层增加 4 字节同步前导码和 2 字节 checksum。
   - 新写入图片在主水印后追加最多 3 份 sync packet 副本。
   - 提取时只有主路径和多尺度候选都失败后，才进入有限 sync packet 搜索。

3. 多尺度候选提取。
   - 覆盖 85% 缩放。
   - 控制候选数量，避免未命中时无限扫描。

4. 音频鲁棒性增强。
   - 采用相对频段能量写入。
   - 保留 legacy QIM 提取回退。
   - 支持 22.05kHz / 44.1kHz 采样率场景和 WAV 重采样回退。

## 关键约束

- 默认提取必须有上限，不能因为未命中而扫完整张图。
- 写入不能显著增加图片处理耗时。
- 任何恢复层都不能破坏主水印层。
- 老版本写出的图片仍必须可提取。
- 未加水印图片的“写入前重复检测”不能被慢路径拖垮。

## 版本策略

当前 payload 仍是 32 字节，不新增字段，不改变 `watermark_uid` 计算，也不影响桌面端和移动端协议。

sync packet 只改变图片水印外层编码，不改变业务 payload 和版权库字段。旧版图片仍可通过主水印层提取；新版图片优先走主水印层，失败后再走 sync packet。

## 当前性能基线

性能基准命令：

```powershell
cargo run --manifest-path watermark-core/Cargo.toml --bin watermark_perf -- --image-dir "E:\Users\jihx\Pictures\人脸" --max-images 3 --max-audio 0
cargo run --manifest-path watermark-core/Cargo.toml --bin watermark_perf -- --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-images 0 --max-audio 1
```

当前实测：

| 媒体 | 操作 | 样本数 | 平均耗时 | 最大耗时 |
| --- | --- | ---: | ---: | ---: |
| 图片 | 写入 PNG | 3 | 4048 ms | 4567 ms |
| 图片 | 原图提取 | 3 | 1172 ms | 1296 ms |
| 图片 | 2% 裁剪提取 | 3 | 41279 ms | 45568 ms |
| 音频 | 30 秒 WAV 写入 | 1 | 2515 ms | 2515 ms |
| 音频 | 30 秒 WAV 提取 | 1 | 646 ms | 646 ms |

产品判断：

- 图片/音频单文件写入可以接受，但批量处理必须走后台队列、进度展示和可取消任务。
- 图片正常提取可以接受。
- 图片裁剪取证提取当前仍偏慢，不适合批量同步执行。
- 裁剪取证应作为“深度检测”路径，UI 上与普通检测区分，并在后续单独优化。

下一步性能优化方向：

1. 对图片裁剪取证做 DWT/YCbCr 候选复用，避免每个 padding 候选重复完整分解。
2. 对 sync packet 提取做 preamble 粗筛，避免每个候选都完整解 38 字节 packet。
3. 批量写入增加并发上限，默认按 CPU 核心数保守并行。
4. 移动端默认只做普通检测，深度裁剪检测由用户显式触发。
