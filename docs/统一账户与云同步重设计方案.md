# HiddenShield 统一账户与云同步重设计方案

## 结论

移动端与桌面端的正式同步方案应改为“统一账户 + 云同步服务”。当前已经实现的“移动端直连桌面端 HTTP 同步”不应作为主产品路径继续打磨，只保留为开发联调、局域网调试或高级导入桥接能力。

正确产品口径：

- 不登录也能本地使用。
- 要跨设备同步，必须登录 HiddenShield 账户并开启云同步。
- 移动端和桌面端都与平台云服务同步，不互相充当对方的服务端。
- 原始媒体和产物媒体默认不上传，首期只同步版权库元数据、取证记录、水印链路、创作者档案、订阅权益和审计记录。

## 为什么要重设计

当前桌面直连方案的问题不是实现细节，而是产品架构方向不对：

- 用户必须同时打开手机和电脑，使用条件脆弱。
- 局域网、防火墙、端口、配对码都会成为同步失败点。
- 没有统一用户身份，无法稳定回答“这条版权记录属于谁”。
- 换机、丢设备、多设备、多桌面、多手机都难以处理。
- 审计、恢复、撤销、团队权限、订阅权益无法自然挂接。
- 移动端和桌面端互相依赖，破坏“本地可独立使用”的产品体验。

因此，直连桌面端只能是调试能力，不是成熟产品同步能力。

## 新产品模型

### 本地模式

用户不登录也可以使用：

```text
移动端 App -> 本地 SQLite
桌面端 App -> 本地 SQLite
```

本地模式支持：

- 图片盲水印写入 / 提取。
- WAV 音频盲水印写入 / 提取。
- 本地版权库。
- 本地取证记录。
- 本地搜索、筛选、详情。

本地模式不支持：

- 跨设备自动同步。
- 换机恢复。
- 云端审计备份。
- 团队共享。

### 云同步模式

用户登录并开启同步后：

```text
移动端 App ─┐
           ├─ HiddenShield Cloud Sync API ─ 账户 / 设备 / 版权库 / 取证 / 审计
桌面端 App ─┘
```

云同步模式支持：

- 移动端和桌面端版权库漫游。
- 创作者身份在移动端和桌面端保持一致。
- 订阅权益在移动端和桌面端保持一致。
- 多设备同步。
- 失败重试。
- 自动解决审计。
- 换机恢复。
- 后续团队空间、云端视频处理。

### 局域网调试模式

当前原桌面直连 HTTP 同步能力保留，但改名和降级：

```text
移动端 App -> 桌面端本地 HTTP 服务
```

用途：

- 开发联调。
- 内部测试。
- 用户高级导入 / 临时迁移。
- 无云环境下的工程验证。

不作为默认入口，不出现在主同步路径里。

## 身份体系

### Account Identity

平台账户身份，是云同步的主归属。

字段建议：

- `account_id`
- `email` / `phone` / 第三方登录标识
- `display_name`
- `created_at`
- `status`

### Workspace

首期可以每个账户默认一个个人空间，后续支持团队。

字段建议：

- `workspace_id`
- `account_id`
- `name`
- `role`
- `created_at`

### Device Identity

每台设备独立生成，不共享。

字段建议：

- `device_id`
- `account_id`
- `device_name`
- `platform`
- `app_version`
- `public_key`
- `last_seen_at`
- `created_at`

### Creator Identity

水印创作者身份，负责水印归属语义。

建议从“本机创作者标签”升级为账户下的创作者档案：

- `creator_id`
- `account_id`
- `workspace_id`
- `display_name`
- `creator_seed_ref`
- `seed_envelope_version`
- `is_default`
- `updated_at`
- `created_at`

成熟产品规则：

- 桌面端和移动端都必须输入的“创作者身份”应统一为账户下的默认创作者档案。
- 继续使用账户并开启云同步后，创作者档案会同步到所有设备，作为水印归属的统一显示和选择来源。
- 设备身份仍然本机独立，不跟随创作者档案同步。
- 首期不上传原始 seed 明文。可以本地保存 seed，云端保存加密包、seed 派生标识或恢复所需的加密 envelope。
- 本地未登录时可使用本机创作者身份；首次登录时提示用户把本机身份加入账户档案，或保留为仅本机身份。

### Entitlement Identity

订阅权益是账户级状态，不属于单台设备。移动端和桌面端都读取同一份权益快照，避免出现“桌面可用、手机不可用”或反向不一致。

首期权益建议从单一“批量处理订阅”扩展为模块化 entitlement：

- `batch_processing`：批量处理能力。
- `cloud_video_processing`：云端视频处理能力，当前规划中。
- `cloud_sync`：云同步容量和设备数。
- `priority_queue`：云端任务优先级，后续可选。
- `team_workspace`：团队空间，后续可选。

字段建议：

- `entitlement_id`
- `account_id`
- `plan_code`
- `status`
- `source`
- `current_period_ends_at`
- `features_json`
- `last_checked_at`
- `updated_at`

客户端只缓存权益快照；最终裁判在云端。

## 客户端同步模式

移动端和桌面端都统一为：

```text
SyncMode.localOnly
SyncMode.cloud
SyncMode.lanDebug
```

### localOnly

- 默认模式。
- 不要求登录。
- 所有记录只保存在本机。
- 同步队列可以存在，但不会发送。

### cloud

- 登录账户后可启用。
- 同步队列发送到云端。
- 拉取云端变更。
- 支持设备游标。

### lanDebug

- 当前桌面 HTTP 同步能力。
- 仅开发 / 高级设置可见。
- 不承诺稳定用户体验。

## 同步数据范围

首期云同步只同步元数据，不同步原始媒体：

- 账户基础信息。
- 默认创作者档案。
- 订阅权益快照。
- 版权记录。
- 水印 UID。
- revision / 写入次数。
- 原始文件 hash。
- 父水印 UID。
- 重写原因。
- 取证记录。
- 提取时间戳。
- 提取设备 ID。
- 提取文件 hash。
- 自动解决审计。
- 客户端队列状态摘要。

默认不同步：

- 原始图片。
- 加水印后的图片。
- 原始音频。
- 加水印后的音频。
- 原始视频。
- 加水印后的视频。
- 本地文件路径。

后续可做可选加密媒体备份，但必须单独授权。

## API 设计

### Auth

```http
POST /v1/auth/continue
POST /v1/auth/refresh
POST /v1/auth/logout
GET  /v1/me
```

`POST /v1/auth/continue` 是登录注册合一入口：

- 如果账号不存在，云端创建账户、默认个人空间、默认创作者档案，并登记当前设备。
- 如果账号已存在，云端直接登录，并 upsert 当前设备信息。
- 客户端不展示单独注册页，也不要求用户理解“注册”和“登录”的区别。
- 该接口必须幂等：同一登录标识重复调用，应返回同一个 `account_id` 和默认 `workspace_id`。

首期可选继续方式：

- 邮箱验证码。
- 手机验证码。
- Apple / Google 后续补。

请求：

```json
{
  "identifier": "alice@example.com",
  "verificationCode": "123456",
  "device": {
    "clientDeviceId": "device_mobile_001",
    "name": "Alice iPhone",
    "platform": "ios",
    "appVersion": "0.1.0",
    "publicKey": "base64..."
  },
  "localCreatorProfile": {
    "displayName": "Alice Creator",
    "creatorSeedRef": "local-seed-ref",
    "seedEnvelopeVersion": 1

}
```

## 自动解决规则

云端应成为正式同步下的最终合并裁判。

建议沿用当前已验证规则：

- 同一 `account_id + source_device_id + client_record_id`：幂等更新。
- 同一 `watermark_uid + asset_fingerprint + revision`：重复忽略。
- 同一 `watermark_uid + asset_fingerprint`，incoming revision 更高：升级版本。
- 同一 `watermark_uid + asset_fingerprint`，incoming revision 更低：忽略旧版本。
- 同一 `watermark_uid`，不同 `asset_fingerprint`：接受为素材变体。

`asset_fingerprint`：

- 优先 `sha256`。
- 取证记录没有 `sha256` 时使用 `extracted_file_hash_hex`。

客户端本地也保留同样规则，用于离线预合并和 UI 去重，但云端裁判结果优先。

## 迁移计划

### Phase 0：冻结决策

- 文档确认：桌面直连不是正式同步方案。
- UI 标注：当前 HTTP 同步改名为局域网调试同步。
- 不继续在直连方案上做产品化投入。

### Phase 1：客户端抽象重命名

- `DesktopPairingProfile -> SyncProfile`
- `SyncTransportMode.http -> lanDebug`
- 设置页把直连同步移到高级区。
- 版权库、本地队列、审计保持不变。

### Phase 2：账户模型

- 增加账户继续态模型。
- 增加默认工作区模型。
- 增加设备注册模型。
- 增加默认创作者档案模型。
- 增加订阅权益快照模型。
- 增加账户设置页。
- 本地记录增加 `account_id` 可空字段。

当前移动端进展：

- `SyncProfile` 已保存 `cloud_base_url / account_id / workspace_id / device_id / creator_profile_id / entitlement_id`。
- 设置页已展示账户、工作区、设备、创作者档案和权益模块摘要。
- `CloudAccountClient` 已实现登录注册合一的 `POST /v1/auth/continue` client。
- `CloudSyncTransport` 已实现 `POST /v1/sync/events:batch` 与 `GET /v1/sync/changes` client。
- `cloud_base_url` 是系统配置提供的客户端地址快照，不属于跨端同步数据；用户设置页不暴露手动填写云服务地址。

当前桌面端进展：

- 已新增 `CloudSyncClient`、`continue_cloud_account`、`push_desktop_vault_record_to_cloud`、`push_saved_desktop_vault_record_to_cloud`、`fetch_cloud_changes`、`flush_desktop_cloud_sync_queue` 命令。
- 已持久化 `cloud_sync_profile.json`，只保存账户 / 工作区 / 设备 / 权益 / token / 游标摘要，不保存原始媒体或本地路径。
- 设置页已提供“账户与云同步”主入口，局域网调试同步降级到高级区。
- 已新增本地 `cloud_sync_queue`，新版权记录成功落库后自动写入云同步队列。
- 版权库页面已展示云同步队列摘要，并支持手动批量补发、单条记录入队同步和拉取云端变更。
- 队列事件只包含版权元数据、hash、水印 UID、revision、父级 UID 和重写原因，不包含原始媒体、产物媒体或本地文件路径。

### Phase 3：云同步 API MVP

- 实现登录注册合一的 `POST /v1/auth/continue`。
- 设备登记随 `auth/continue` 自动完成。
- 实现创作者档案 upsert / pull。
- 实现权益快照 pull。
- 实现 push batch。
- 实现 pull changes。
- 云端实现自动解决审计。

### Phase 4：双端接入

- 移动端接 `CloudSyncTransport`。
- 桌面端接 `CloudSyncTransport`。
- 局域网调试同步隐藏到高级设置。

### Phase 5：迁移与回滚

- 本地旧记录首次继续使用账户后提示“加入云同步”。
- 用户可选择：
  - 仅本机保留。
  - 上传元数据到云端。
  - 导出本地备份。
- 云同步失败不影响本地使用。

## 风险

- 云同步需要后端、账号、鉴权和运维，工作量明显大于直连。
- 隐私政策和用户协议必须更新。
- 订阅权益和云同步容量要提前定义。
- 如果云服务不稳定，会影响跨设备体验，所以必须保证本地模式完全可用。

## 推荐下一步

先做 Phase 3：云同步 API MVP。优先定义并实现登录注册合一的 `POST /v1/auth/continue`，让它同时完成账户 upsert、默认工作区返回、设备登记、创作者档案 upsert 和权益快照返回，再接版权库 push / pull changes。这样移动端现有 `SyncProfile` 契约可以直接对接真实服务端。
