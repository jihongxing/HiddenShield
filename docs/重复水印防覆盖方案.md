# 重复水印防覆盖方案

## 结论

默认不允许对已含有隐盾水印的图片或音频再次写入。

如果业务上确实需要二次写入，必须显式选择“允许重写”，并在金库里记录继承关系，而不是把“第几次写入”塞进水印本体。

---

## 为什么不把次数写进水印

这种做法看起来直接，实际上不可靠。

1. 次数字段可以被第二次写入者伪造。
2. 提取结果只能说明“当前文件里有什么”，不能证明历史顺序。
3. 一旦发生转码、压缩、裁剪，内部次数和真实链路更容易失真。

所以，“次数”可以做展示字段，不能做版权真相。

---

## 推荐方案

### 方案 A: 只加次数字段

不推荐。

优点：

- 实现快。

缺点：

- 不可信。
- 容易被伪造。
- 不能防止覆盖。

### 方案 B: 写入前探测，发现已有水印就拒绝

推荐。

优点：

- 最简单。
- 直接保住第一次写入。
- 图片和音频可以共用同一套路。

缺点：

- 不能默认支持覆盖式二次处理。

### 方案 C: 写入前探测 + 显式允许重写 + 金库记录父子关系

推荐作为 B 的补充。

优点：

- 既能保护首个版权记录，也保留合法派生链路。
- 后续可在 UI 中展示“这是第几次派生处理”。

缺点：

- 比 B 多一层数据库字段和界面文案。

---

## 当前实现

### 图片

- `embed_image_watermark_bytes()` 默认先提取，若已有水印则返回 `AlreadyWatermarked`
- `embed_image_watermark_bytes_allow_rewrite()` 允许显式覆盖
- 路径版 `embed_image_watermark()` 也遵守同样规则

### 音频

- `embed_watermark_samples()` 默认先提取，若已有水印则返回 `AlreadyWatermarked`
- `embed_watermark_samples_allow_rewrite()` 允许显式覆盖
- WAV 字节版同样遵守同样规则

### 服务层

`WatermarkService::embed(..., EmbedOptions { allow_rewrite: false })` 为默认安全路径。  
只有显式把 `allow_rewrite` 设为 `true`，才会走覆盖式写入。

### 金库继承链

`vault_records` 已增加三个字段：

- `parent_watermark_uid`：父级水印 UID，首写记录为空。
- `revision`：写入版本，首写为 1。
- `rewrite_reason`：显式重写原因，普通首写为空。

当前默认流程仍然只生成 `revision = 1` 的首写记录。后续 UI 如果开放“允许重写”，必须写入父级 UID 和重写原因。

### 显式重写入口

工作台已为图片和音频开放高级选项“允许重写已有隐盾水印”。

启用后：

1. 写入前先提取当前文件中的父级水印 UID。
2. 使用 `allow_rewrite = true` 显式覆盖媒体内水印。
3. 如果本地金库能找到父级记录，新记录的 `revision = parent.revision + 1`。
4. 新记录保存 `parent_watermark_uid` 和 `rewrite_reason`。
5. 开始任务前弹出风险确认，说明旧水印会被覆盖，金库会记录父子链路。

视频压制流程仍然不开放重写入口，避免把“源视频二次覆盖”和“平台输出压制”混在一起。

---

## 后续建议

1. 显式重写确认弹窗在任务开始前先提取父级 UID，并在弹窗里展示出来。
2. 取证结果页展示父子链路。
3. 默认拦截所有重复写入，避免用户无意覆盖第一次证据。
4. 对图片和音频统一使用同一套规则，不做两个标准。
