# 移动端规划 (Mobile Roadmap)

## 技术基础

Tauri 2 原生支持 Android + iOS，现有架构可复用：
- Vue 前端 → WebView 直接渲染
- Rust 水印算法 → 交叉编译到 ARM，纯计算无平台依赖
- SQLite → rusqlite 支持移动端
- TSA 网络请求 → reqwest 支持移动端

## 产品定位：轻量级移动哨站

移动端不是桌面端的"缩小版"，而是**随身版权哨站**：
- 随拍随护：拍完照片/录完播客，秒级本地确权
- 重型资产上云：视频压制交给云端 GPU 集群
- 金库漫游：桌面端和移动端版权记录互通

## 移动端三大致命陷阱

### 1. iOS 进程唤起禁令

桌面端依赖 `tokio::process::Command` 动态唤起 FFmpeg 子进程。iOS 沙盒**绝对禁止**运行时执行外部二进制（`Operation not permitted`）。

唯一解法：将 FFmpeg 编译为静态库 (.a)，通过 C FFI 绑定进 Rust。代价是包体积 +50MB，且 C 内存泄漏极易崩溃。

**结论：移动端不跑 FFmpeg，视频走云端。**

### 2. 算力与续航悖论

即使最新 iPhone，纯 CPU 压制 10 分钟 4K 视频会在 2 分钟内发烫降频。移动 OS 的温控监控会触发 OOM 或 Thermal Kill，直接强杀 App，进度条永远停在 40%。

**结论：移动端只做轻量计算（图片 DWT-DCT-SVD、音频 QIM），绝不做视频编解码。**

### 3. 后台生存权

用户点击"开始处理"后切到微信，几秒后 CPU 时间片被剥夺。桌面端的 Wake Lock 策略在移动端无效，切后台即断气。

**结论：移动端任务必须秒级完成（图片/音频），或走云端异步回调（视频）。**

## 分阶段实施

### Phase 1：图片水印 + 取证（最小可行移动端）

- DWT-DCT-SVD 图片盲水印嵌入/提取
- 创作者身份系统
- 本地版权金库
- 维权取证 + 存证报告
- 预计包体积：< 10MB

### Phase 2：音频水印

- QIM 频域盲水印（realfft 跨平台）
- 音频文件选择器适配移动端沙盒

### Phase 3：视频处理（云端）

- 本地不跑 FFmpeg，视频压制走云端 API
- 移动端并发上传视频 → 云端 K8s + GPU 集群压制 + 嵌入水印 → 推送通知 + 下载成品
- 手机端只负责进度回调和下载
- 作为 Pro 移动端功能

### Phase 4：金库数据漫游

- 基于 User Seed 打通桌面端和移动端 SQLite 版权金库
- 端到端加密同步（不暴露原文件内容）
- 电脑上处理的视频，手机上随时查阅证书和维权报告

## 平台差异与适配

| 问题 | Android | iOS |
|------|---------|-----|
| FFmpeg | 需预编译 .so (arm64) +30MB | 禁止动态加载，必须静态链接或用 AVFoundation |
| 文件访问 | SAF (Storage Access Framework) | Document Picker |
| 硬件编码 | MediaCodec API | VideoToolbox |
| 防休眠 | WakeLock（受限） | beginBackgroundTask（最多 30s） |
| 后台任务 | WorkManager | BGTaskScheduler |
| 温控强杀 | Thermal throttling → ANR | Thermal Kill → 进程终止 |
| 分发 | Google Play / APK 直装 | App Store（审核严格） |

## 移动端不做的事

- ❌ 本地视频压制（算力不够、会被系统杀）
- ❌ 本地 FFmpeg 调用（iOS 禁止子进程）
- ❌ 硬件编码检测（移动端统一走云端）
- ❌ 多平台并行输出（移动端单任务即可）
- ❌ 长时间后台任务（切后台即死）

## UI 适配要点

- 响应式布局：侧边栏改为底部 Tab
- 拖拽改为系统文件选择器 / 相册选择
- 进度通知：利用系统通知栏 + 推送
- 分享：存证报告直接调用系统分享面板
- 相机直连：拍照后一键确权

## 时间线预估

- Phase 1：2-3 周（核心逻辑已有，主要是 UI 适配 + 移动端构建配置）
- Phase 2：1 周
- Phase 3：需要云端基础设施就绪后再启动
- Phase 4：需要账号体系 + 同步协议设计
