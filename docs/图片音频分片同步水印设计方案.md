# 图片/音频分片同步水印设计方案

## 背景

当前图片水印已经具备主水印层和 sync packet 备份层，真实素材 2% 裁剪取证提取已经从 41 秒级优化到约 1.3 秒。音频水印采用相对频段能量写入，已支持音量变化、低采样率和 WAV 重采样回退。

新的算法设想是：把图片或音频分成多个切片，在每个切片写入短同步标记；提取时先扫同步标记，命中后再进入完整 payload 解析。这个方向可行，本质上是给盲水印增加“定位索引层”。

需要注意的是，这里的“特定字符”不应使用普通文本字符，而应设计成二进制同步标记。同步标记需要包含前导码、版本、分片编号、payload 绑定信息和校验码，避免误命中。

音频侧的产品边界已经收敛为：只为 30 秒及以上音频生成版权保护副本。5 秒 / 10 秒短裁剪可继续作为取证观察项，但不作为首版版权保护承诺，也不再驱动当前生产算法继续深挖。

## 目标

1. 提升裁剪、截取、局部损坏后的定位效率。
2. 提升局部可验证性：只保留部分图片或部分音频时，仍有机会发现水印。
3. 保持当前主水印层和 sync packet 兼容，不破坏旧素材提取。
4. 为桌面端和移动端的“快速检测 / 深度检测 / 批量检测”提供更稳定的底层能力。

## 不做的事

- 不把 16 个切片都写入完整业务 payload。
- 不替换当前主水印层。
- 不改变 `WatermarkPayload` 的 32 字节业务格式。
- 不在 UI 暴露 DWT、SVD、FFT、sync marker 等技术词。
- 不承诺短于 30 秒的音频可完成版权保护。

## 总体方案

采用三层结构：

1. 主水印层。
   - 继续写入完整 payload。
   - 保持老图、老音频兼容。

2. 全局取证层。
   - 图片保留当前 sync packet。
   - 音频保留当前相对频段完整 payload 投票。

3. 分片同步索引层。
   - 图片按空间分片。
   - 音频按时间分片。
   - 每个分片只写短 marker。
   - marker 命中后，再进入对应候选区域的完整解析。

## Marker 结构

建议 marker 长度控制在 8 字节：

| 字段 | 长度 | 说明 |
| --- | ---: | --- |
| preamble | 2 bytes | 固定同步头，例如 `0xA7 0x5C` |
| version | 1 byte | marker 版本 |
| slice_id | 1 byte | 图片 tile 编号或音频时间片编号 |
| payload_tag | 2 bytes | payload 哈希短标签 |
| checksum | 2 bytes | 对前 6 字节的校验 |

`payload_tag` 应由完整 payload 的稳定哈希截断得到，用于把 marker 与完整水印绑定。这样攻击者即使复制某个 marker，也难以伪造完整 payload 的一致性。

## 图片分片设计

### 分片方式

默认采用 `4x4 = 16` 个 tile。

每个 tile 内写入一个短 marker。marker 不写在 tile 的极端边缘，而写在 tile 内部的稳定区域，避免轻微裁剪直接损坏所有同步信息。

### 提取流程

1. 先运行当前快速主水印提取。
2. 失败后尝试当前 2% 裁剪 sync packet 高频路径。
3. 若仍失败，进入分片 marker 扫描。
4. 只读取每个 tile 的短 marker。
5. 命中 marker 后，根据 `slice_id` 和 `payload_tag` 选择对应候选区域做完整 payload 或 sync packet 解析。
6. marker 全部未命中，再进入现有兜底候选。

### 预期收益

- 裁剪后如果仍保留多个 tile，可快速定位到未损坏区域。
- 局部截图取证更有机会命中。
- 未命中时只读取短 marker，比完整候选解析成本低。

### 风险

- 图片缩放会改变 tile 边界。
- DWT 不是平移不变的，过度依赖固定 tile 坐标会降低鲁棒性。
- tile marker 写入过多会增加写入耗时和视觉扰动风险。

因此图片方案不能只依赖绝对 tile，必须保留当前 padding/scale 兜底机制。

## 音频分片设计

### 分片方式

音频天然适合按时间分片。建议把可用帧分为 16 个时间片，每个时间片写入短 marker。

对 30 秒音频，16 个时间片约等于每片 1.875 秒。实际实现应按 FFT frame 数分片，而不是按秒硬切。

### 提取流程

1. 先运行当前完整 payload 投票提取。
2. 失败后进入时间片 marker 扫描。
3. 每个时间片只解析短 marker。
4. marker 命中后，在该时间片及相邻时间片做完整 payload 聚合。
5. 对截取音频，允许作为取证输入尝试局部恢复，但不承诺任意短片段都能恢复完整 payload。

### 预期收益

- 对音频截取更有帮助。
- 对长音频批量检测更容易做早停。
- 对移动端检测更友好，可以边扫边反馈进度。

### 风险

- 当前音频 payload 是跨全部 frame 投票，截取后可能仍要重新设计局部聚合规则。
- marker 写入太强会产生可听失真，太弱又容易被 MP3 转码破坏。
- 静音片段不适合写 marker，需要跳过或降级。

## 效率判断

分片同步层的效率收益来自两点：

1. marker 比完整 payload 短。
2. 命中 marker 后可以缩小完整解析范围。

如果 marker 仍然使用和完整 payload 一样的昂贵读取方式，收益会有限。因此原型验证必须分别测量：

- marker 扫描耗时。
- marker 命中率。
- marker 命中后完整 payload 恢复耗时。
- 对原图/原音频快速路径的影响。

## 兼容策略

写入：

- 新版本素材写入主水印层、现有取证层和分片 marker 层。
- 旧版本素材不含分片 marker，仍按当前路径提取。

提取：

- 先走当前快速路径。
- 再走高概率裁剪/截取路径。
- 再走分片 marker。
- 最后走现有兜底。

版权库：

- `watermark_uid` 不变。
- 不新增用户可见字段。
- 可在内部取证结果里记录命中的 `slice_id` 和检测方式，便于后续审计。

## 原型验证计划

### P1：文档和工具验证

先新增一个实验工具，不直接改变生产写入格式：

```powershell
cargo run --manifest-path watermark-core/Cargo.toml --bin slice_marker_probe -- --image-dir "E:\Users\jihx\Pictures\人脸" --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-images 3 --max-audio 1
```

工具目标：

- 在不改生产算法的情况下，模拟 16 分片扫描成本。
- 对图片输出每个 tile 的 marker 读取成本估算。
- 对音频输出每个时间片的 marker 读取成本估算。
- 给出是否值得进入生产实现的判断。

当前实验工具已新增：

- `watermark-core/src/bin/slice_marker_probe.rs`

首次真实素材测量：

```powershell
cargo run --manifest-path watermark-core/Cargo.toml --bin slice_marker_probe -- --image-dir "E:\Users\jihx\Pictures\人脸" --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-images 3 --max-audio 1
```

结果：

| 媒体 | 样本数 | 扫描方式 | 耗时 |
| --- | ---: | --- | ---: |
| 图片 | 3 | 4x4 tile marker 成本估算 | 1-2 ms / 张 |
| 音频 | 1 | 16 time marker 成本估算 | 219 ms / 30 秒音频 |

判断：

- 图片 marker 扫描成本极低，作为索引层可行。
- 音频 marker 扫描成本低于当前完整 30 秒音频提取，可作为截取取证和早停检测的候选方案。
- 当前工具只是扫描成本估算，尚未验证 marker 写入后的鲁棒性；后续音频优化应服务 30 秒以上作品的稳定写入和快速取证，不继续以 5 秒 / 10 秒片段恢复作为当前主目标。

### P2：图片最小实现

- 在图片 sync packet 后追加 16 个 tile marker。
- 提取时先读 tile marker，再按命中 tile 解析。
- 验证原图、2% 裁剪、局部裁剪、85% 缩放、JPEG Q75。

### P3：音频最小实现

- 在每个时间片写短 marker。
- 截取样例验证 marker 命中；该项属于研究验证，不作为当前产品承诺。
- 验证音量变化、MP3 192k 转码、低采样率。

当前实现进展：

- 已在 `watermark-core/src/audio.rs` 增加 16 时间片 marker 原型。
- marker 为 8 字节短包：`preamble + version + slice_id + payload_tag + checksum`。
- 30 秒以上音频启用 marker 索引层；短于 30 秒的音频由产品入口拒绝写入。
- marker 使用多 bit / 多 lane 写入：每帧写多个 marker bit，每个 bit 由多个频段对投票。
- 完整 payload 仍保留原有相对频段投票；marker 作为辅助索引层，不替代主水印。

### 音频截取验证结果

已新增单测验证：

- 从已加水印长音频样本中截取中间片段。
- 截取片段仍可检测到 audio slice marker。
- 新增真实素材 bit error 诊断工具：`watermark-core/src/bin/audio_recovery_diag.rs`。
- 真实 ffmpeg `clip_10s_middle` 已重新纳入 `robustness_bench`。

结论：

- 分片 marker 对截取场景有效，能作为“这段音频来自已加水印素材”的定位索引。
- 已补上“音频局部恢复包”：每个长音频周期会滚动重复写入一个完整 recovery packet，截取片段可直接恢复完整 payload。
- 真实 ffmpeg 中段截取的根因已经定位：原始 recovery 读法只看单个重复周期，且每个 bit 只有 2 个 lane，真实裁剪后常残留约 20 个 bit 错误。
- 当前生产修复采用三点组合：跨重复周期聚合投票、recovery bit lane 从 2 提升到 3、只在 preamble 接近命中时启用最多 3 个 payload bit 的受限纠错。
- 纠错必须同时通过 recovery checksum 和 payload HMAC，避免把随机噪声误判为水印。

最新验证：

```powershell
cargo test --manifest-path watermark-core/Cargo.toml
cargo run --manifest-path watermark-core/Cargo.toml --bin audio_recovery_diag -- --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-audio 1 --phase-step 512 --top 5
npm run watermark:bench -- --image-dir "E:\Users\jihx\Pictures\人脸" --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-images 3 --max-audio 1
```

结果：

| 项目 | 结果 |
| --- | ---: |
| watermark-core 全量单测 | 23/23 passed |
| 真实素材鲁棒性 | 25/25 passed |
| embedded_30s recovery bit errors | 0 |
| clip_10s_middle recovery bit errors | 5，受限纠错后可提取 |

阶段判断：

- 图片分片/sync 方案已经具备产品化基础。
- 音频分片 marker 和 recovery 方案已经从原型进入阶段稳定状态。
- 当前稳定性结论基于 1 个真实 mp3 样本和合成单测，下一阶段仍应扩展到更多歌曲、不同码率和不同裁剪长度。

### 音频裁剪矩阵扩展验证

已在 `robustness_bench` 增加可选重型矩阵：

```powershell
npm run watermark:matrix -- --image-dir "E:\Users\jihx\Pictures\人脸" --audio-glob "E:\Users\jihx\Pictures\*.mp3" --max-images 1 --max-audio 1
```

矩阵内容：

- WAV 裁剪：5s / 10s / 15s × start / middle / end。
- MP3 192k roundtrip 后裁剪：5s / 10s / 15s × start / middle / end。
- 默认 `watermark:bench` 不启用该矩阵，避免日常验证过慢。

当前 1 图 + 1 音频真实素材结果：

| 模式 | 结果 |
| --- | ---: |
| 默认 bench | 13/13 passed |
| audio matrix | 23/31 passed |

失败集中在：

- 5 秒中段 / 结尾裁剪。
- 10 秒结尾裁剪。
- 15 秒中段裁剪。
- MP3 roundtrip 后同类短裁剪场景。

判断：

- 10 秒中段裁剪已经可作为扩展取证观察样例保留在默认 bench。
- 更短、更靠近结尾、以及 MP3 后再裁剪的组合仍未达标，不能直接宣称“任意 5 秒可恢复完整 payload”。
- 已尝试“虚拟 recovery 包相位偏移”方案，结果导致 `clip_10s_middle` 回归失败，已撤回。下一轮应从写入层增强入手，例如缩短 recovery 包、增加更强 ECC，或做交错式多起点 recovery 写入，而不是继续扩大提取端猜测范围。

### Recovery v2 实验记录

已尝试两种直接改生产 recovery 布局的方案，均已撤回：

1. 压缩单包周期。
   - 尝试把 `AUDIO_RECOVERY_BITS_PER_FRAME` 从 18 提升到 34，并把 lane 从 3 降到 2。
   - 结果：默认 bench 退化，`clip_10s_middle` 从 PASS 变为 FAIL。
   - 再尝试温和版本 24 bit/frame + 3 lane，仍导致默认 bench 退化。
   - 判断：当前频段容量下，直接压缩单包周期会降低每 bit 稳定性，不适合作为小改动上线。

2. 同频段交错起点。
   - 尝试在奇偶 recovery 周期使用不同半包偏移，避免额外占用频段。
   - 结果：合成裁剪 recovery 单测失败。
   - 根因：裁剪后丢失原始 recovery 周期编号，提取端无法可靠知道当前帧属于哪个交错起点。
   - 判断：交错写入必须携带可恢复的局部同步信息，不能只依赖原始全局 frame index。

当前结论：

- 保留稳定版 recovery：18 bit/frame、3 lane、跨重复周期聚合、受限纠错。
- `--audio-matrix` 作为扩展评估工具保留，不把 23/31 当成失败。
- recovery v2 需要重新设计小型局部同步头或短包 ECC，但当前阶段先归档，不进入生产写入。
- 详细 V2 方案见：`docs/音频RecoveryV2短包纠错设计.md`。

## 验收标准

图片：

- 原图提取保持 1 秒级。
- 2% 裁剪提取不退化，保持 2 秒以内。
- 局部裁剪命中率高于当前方案。
- 真实素材鲁棒性不低于当前 24/24。

音频：

- 音频写入要求原始音频至少 30 秒。
- 30 秒 WAV 提取保持 1 秒级。
- 10 秒中段裁剪保留为扩展取证观察项，不作为产品承诺。
- 5 秒裁剪、结尾裁剪和 MP3 后再裁剪进入扩展矩阵跟踪，暂不作为已达标能力宣传。
- MP3 转码后 marker 不应大面积失效。

## 结论

分片同步水印可行，但它应该作为“索引层”加入现有算法，而不是替代主水印层。图片侧要谨慎处理缩放和裁剪导致的 tile 边界漂移；音频侧已经进入 30 秒以上作品保护边界，短片段只作为取证辅助观察，不作为当前生产承诺。

推荐落地顺序：

1. 保留实验工具验证 marker 扫描成本。
2. 优先完善 30 秒以上音频写入稳定性、回读验证和用户提示。
3. 再推进图片 4x4 tile marker 原型。
